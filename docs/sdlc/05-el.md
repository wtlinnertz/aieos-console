## 3. Experiment Inventory

### EXP-1: Manual Process Barrier Assessment

| Field | Value |
|-------|-------|
| Target Assumption | ASM-1 (Manual process is the primary adoption barrier) |
| Hypothesis Tested | That the manual process of navigating kit files, sequencing artifacts, and tracking state is the primary barrier preventing AIEOS adoption |
| Method | Direct observation during framework operation: initiative owner used AIEOS manually (early CLI test via EEK) and is currently using it for this initiative (full PIK flow) |
| Sample / Scope | Single user (initiative owner), two engagements: one prior EEK test, one current PIK flow |
| Conclusion | Partially Confirmed |
| Confidence Level | Low |

**Raw Findings:**
- The initiative owner completed an early end-to-end EEK test to build a simple CLI, reporting that the artifact flow works but the manual process is labor-intensive
- The current PIK engagement (this initiative) required reading 4 files per artifact type (spec, template, prompt, validator) plus the playbook, principles files, and worked examples before each generation step
- The initiative owner's decision to build aieos-console was explicitly motivated by the difficulty of broader adoption without a guided interface
- No other users have attempted to use AIEOS, so the "primary barrier" claim has not been tested with external users — it is possible that other barriers (framework trust, perceived complexity of the governance model itself, organizational fit) outweigh the manual process barrier

**Limitations:**
- Single-user observation — the initiative owner has deep familiarity with the framework, which may bias assessment of what is "difficult" for others
- No external user data — the "primary barrier" claim is extrapolation from one user's experience
- Self-selection bias — the initiative owner chose to build a console, which does not prove that other potential adopters would identify the same barrier

### EXP-4: Validator Quality Signal Assessment

- Small sample size (4 artifacts)
- No deliberately bad artifacts tested — validators were not tested against known-bad inputs to verify they correctly reject poor quality

### EXP-5: Interaction Model Preference

| Field | Value |
|-------|-------|
| Target Assumption | ASM-5 (A guided interface is the right interaction model) |
| Hypothesis Tested | That a step-by-step guided interface is the right model for AIEOS process execution |
| Method | Observation during this PIK engagement — the current session itself is a manual walkthrough of the guided process (WCR → Intake → PFD → VH → AR → EL), approximating what a guided interface would do |
| Sample / Scope | Single user (initiative owner), one engagement |
| Conclusion | Partially Confirmed |
| Confidence Level | Low |

**Raw Findings:**
- The current session followed a strict linear sequence (WCR → Intake → PFD → VH → AR → EL) and the initiative owner approved each step before proceeding — this sequential, guided pattern worked for the discovery flow
- The initiative owner did not attempt to skip steps, reorder artifacts, or jump ahead — the linear flow was accepted without friction
- At no point did the initiative owner express a preference for a different interaction model (chat-only, IDE plugin, etc.)
- However, this was a single first-time walkthrough — experienced users may find a rigid sequential model too constraining after familiarity develops

**Limitations:**
- Single user, single session — cannot generalize to other users or repeated usage
- The initiative owner was predisposed to accept a guided model (they proposed it)
- No alternative models were presented for comparison
- The current session is conversational (chat-based), not a wizard UI — it does not fully test the wizard interaction model

## 6. Impact Assessment

### Invalidated Assumptions

No assumptions were invalidated.

### Partially Confirmed Assumptions

**ASM-2: Non-Technical Users Want Independent Operation**
- **What was confirmed:** A product manager expressed a general preference for self-service tooling, consistent with the assumption that PMs would want independent access
- **What was not confirmed:** Whether this preference holds specifically for AIEOS processes, which are more structured and governance-heavy than typical PM tooling
- **Impact on initiative:** Sufficient directional signal to include UG-2 features in scope. AIEOS-specific validation should continue during development.
- **Affected hypotheses:** VH HYP-2 (non-technical independent operation)

**ASM-1: Manual Process Is the Primary Adoption Barrier**
- **What was confirmed:** The manual process is labor-intensive and creates real friction for framework operation
- **What was not confirmed:** Whether the manual process is the *primary* barrier for external adopters (vs. framework trust, governance model complexity, organizational fit)
- **Impact on initiative:** The initiative remains viable — the confirmed portion (manual friction exists) justifies a guided interface. The unconfirmed portion (primary barrier) means adoption impact may be smaller than hypothesized if other barriers dominate.
- **Affected hypotheses:** VH HYP-1 (guided sequencing), VH HYP-4 (evaluation barrier)

**ASM-4: Prompts Work Across LLM Providers**
- **What was confirmed:** Prompts produce passing artifacts with at least one provider (Claude)
- **What was not confirmed:** Cross-provider performance — the core claim of the assumption
- **Impact on initiative:** The LLM configurability feature is viable in principle but the cross-provider value proposition remains unvalidated. If prompt output quality varies significantly across providers, the console may need provider-specific prompt tuning, which adds complexity.
- **Affected hypotheses:** VH HYP-3 (integrated validation improves quality — assumes reasonable first drafts from prompts)

**ASM-5: Guided Interface Is the Right Interaction Model**
- **What was confirmed:** A sequential, step-by-step flow works for at least one user on first use
- **What was not confirmed:** Whether this model is preferred over alternatives, or whether it scales to repeated use
- **Impact on initiative:** Low risk — the sequential model is a reasonable starting point, and the console can evolve its interaction model based on user feedback
- **Affected hypotheses:** VH HYP-1 (guided sequencing), VH HYP-2 (non-technical independence)

**ASM-6: Validators Provide Quality Signal**
- **What was confirmed:** Validators correctly identified one real quality issue (no_solutions violation) and correctly passed three well-formed artifacts
- **What was not confirmed:** Validator performance against poor-quality inputs, and validator accuracy at scale
- **Impact on initiative:** Moderate confidence that the validation integration feature will work. The one-FAIL-correctly result is encouraging. Further testing with deliberately varied-quality inputs would strengthen confidence.
- **Affected hypotheses:** VH HYP-3 (integrated validation improves quality)

### Confirmed Assumptions (De-Risk Notes)

No assumptions were fully confirmed. All tested assumptions are partially confirmed with low-to-medium confidence.

## 7. Recommendations

**Proceed / Pivot / Pause:** Proceed with acknowledged risks

**Rationale:**
- No assumptions were invalidated — no evidence contradicts the initiative's direction
- The partially confirmed results for ASM-1, ASM-4, ASM-5, and ASM-6 provide enough directional confidence to proceed to DPRD generation
- The inconclusive result for ASM-2 (non-technical user demand) is a known risk but does not block the initiative — the console serves UG-1 (technical users) independently of whether UG-2 (non-technical users) wants self-service access
- The most significant unvalidated risk is ASM-1's "primary barrier" claim for external adopters — but even if other barriers exist alongside the manual process barrier, reducing manual friction has standalone value for existing users
- Cross-provider LLM testing (ASM-4) can be conducted during development without blocking requirements definition

**Remaining validation needed:**
- ASM-2 (non-technical user demand) has directional support from PM feedback but should be validated with AIEOS-specific user testing during development
- ASM-4 (cross-provider LLM performance) should be tested with at least one additional provider during development. The system will be architected for multi-provider support regardless.
- ASM-1 (primary barrier for external adopters) should be validated via adopter interviews when candidates are available. This affects expected adoption impact but does not change core product direction.
