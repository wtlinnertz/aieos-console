## 3. Exposure Specification

**Exposure strategy:** immediate full — single-user local Docker deployment

**Exception justification:** RCF §3 documents that progressive exposure is not applicable to single-user local Docker deployments. RCF §7 Exception Table explicitly states: "Single-user local Docker deployment makes progressive exposure technically impossible — one container, one user, no traffic splitting." There is one container serving one user; exposure is binary (deployed or not deployed). The two-stage requirement from rp-spec.md §Exposure Specification is satisfied by the pre-deployment verification stage and the post-deployment watch stage below.

| Stage | Target | Trigger to Proceed | Hold Action if Trigger Not Met |
|-------|--------|-------------------|-------------------------------|
| Stage 1: Pre-deployment verification | 0% (not yet exposed) | Verification script (`scripts/verify-docker.sh`) passes all checks: Docker build, health check, project initialization, state persistence | Do not proceed to exposure; investigate failure; re-run verification after fix |
| Stage 2: Full exposure + watch period | 100% (single user) | 1-hour watch period complete; all success criteria met (§6); zero application errors; at least one successful LLM generation and one successful LLM validation completed | Trigger rollback per §4 if rollback conditions met; extend observation if inconclusive |

**Feature Flag Configuration:**

No feature flags are used in this release. This is the initial release — all functionality is enabled by default. RCF §3 permits up to 3 simultaneous active flags; none are needed for this release.

**RCF Compliance:**

## 4. Rollback Specification

**Rollback Conditions:**

Rollback must be triggered when any of the following conditions is observed:

| Condition | Signal | Threshold | Sustained Duration |
|-----------|--------|-----------|-------------------|
| 1 — Application error | Application returns HTTP 500 errors on core endpoints (`/api/health`, `/api/project`, `/api/kits`) | Any HTTP 500 response on health check; or ≥2 HTTP 500 responses on other core endpoints | Any single health check failure; 2 consecutive failures on other endpoints |
| 2 — Health check failure | Docker HEALTHCHECK reports unhealthy (`docker inspect --format='{{.State.Health.Status}}'`) | Status = `unhealthy` | Container marked unhealthy by Docker (3 consecutive failures per HEALTHCHECK retries config) |
| 3 — LLM API integration failure | LLM API calls fail with non-transient errors (authentication failure, invalid model, persistent timeout) | All LLM API calls fail during the watch period (0% success rate after initial attempt) | 3 consecutive LLM API call failures with non-transient error codes |
| 4 — State corruption | `.aieos/state.json` becomes unreadable or inconsistent after a write operation | State file fails to parse or state metadata contradicts artifact files | Any single occurrence |

**Rollback Procedure:**

Reference: ORD §8 Rollback procedure (TDD §9 Rollback: 4-step procedure)

Summary:
1. Stop the running container: `docker stop aieos-console && docker rm aieos-console` (ORD §8 Rollback step 1)
2. For initial release: no previous version exists to roll back to. If the release fails, the container is stopped and the release is declared failed. Project data in the mounted volume is preserved and accessible as files. (ORD §8 Rollback step 2 — "run previous image tag" is not applicable for first release)
3. Verify project directory integrity: confirm `.aieos/state.json` and any generated artifact files are intact and readable in the project directory (ORD §8 Rollback step 3)
4. Document the failure in the Release Record with root cause and remediation plan (ORD §8 Rollback step 4)

**Responsible Party:** T. Owner — backup: N/A (single-person initiative; if T. Owner is unavailable, release is paused until available)

**Decision Authority:** T. Owner — escalation if unavailable: release is paused (no escalation path for single-person initiative; the release window is flexible with no external dependencies)

**Decision Window:** Rollback decision must be made within 15 minutes of a condition being triggered. For this single-user local deployment, the release owner is actively monitoring during the entire watch period, so the decision window is effectively immediate. If no decision is made within 15 minutes, rollback is initiated automatically (stop container).

## 6. Post-Release Monitoring Plan

**Metrics to Watch:**

| Metric Name | Baseline Value (from ORD) | Alert Threshold | Monitoring System |
|-------------|--------------------------|-----------------|------------------|
| `health_check_response` — HTTP status from `/api/health` | HTTP 200 (ORD §3.10, §5.5) | Any non-200 response | `curl -f http://localhost:3000/api/health` (manual or scripted polling) |
| `docker_health_status` — Docker container health status | `healthy` (ORD §3.13 HEALTHCHECK config) | Status = `unhealthy` | `docker inspect --format='{{.State.Health.Status}}' aieos-console` |
| `llm_generation_duration` — Time for LLM artifact generation call | Baseline not yet established — first release | >120 seconds (RCF §6 absolute threshold for first release) | LLM usage record in `.aieos/state.json` (`duration` field) |
| `llm_validation_duration` — Time for LLM validation call | Baseline not yet established — first release | >120 seconds (RCF §6 absolute threshold for first release) | LLM usage record in `.aieos/state.json` (`duration` field) |
| `application_error_count` — Count of HTTP 500 responses during watch period | 0 (ORD §4.7, RCF §6 absolute threshold) | >0 unexpected errors | Docker logs (`docker logs aieos-console`) — structured JSON error events |
| `llm_api_failure_count` — Count of failed LLM API calls | 0 (first release absolute threshold per RCF §6) | >0 non-transient failures (auth errors, invalid model) | Docker logs — `llm.generation_failed` structured log events |
| `state_persistence_intact` — state.json readable after container restart | Verified (ORD §3.8) | state.json missing or unparseable after restart | Manual check: `docker restart aieos-console` then `curl http://localhost:3000/api/project` |

**Watch Period:** 1 hour after deployment — meets RCF §6 minimum of 1 hour

**Check-In Schedule:** Release owner checks metrics every 15 minutes during the 1-hour watch period (4 check-ins total: T+15min, T+30min, T+45min, T+60min). Each check-in: verify health endpoint, review Docker logs for errors, confirm container health status.

**Success Criteria (proceed from Stage 2 to release complete):**

All of the following must be true for the full 1-hour observation period before declaring the release successful:
- Health check returns HTTP 200 on every check-in (`/api/health`)
- Docker container health status remains `healthy` throughout
- Zero unexpected application errors in Docker logs
- At least one successful LLM generation completed (per RCF §6 Tier 1 tightening)
- At least one successful LLM validation completed (per RCF §6 Tier 1 tightening)
- State persistence verified: container restart preserves `.aieos/state.json`
