## 2. Upstream References

**Problem Framing Document:** PFD-CONSOLE-001 — AIEOS requires manual navigation of 6 kits and 33+ artifact types with no guided interface, state tracking, or automation, creating a high barrier to entry for all users and excluding non-technical users entirely.

**Value Hypothesis:** VH-CONSOLE-001 — Four hypotheses: guided sequencing reduces errors/time (HYP-1), non-technical users can operate independently (HYP-2), integrated validation improves artifact quality (HYP-3), reduced evaluation barrier enables adoption decisions (HYP-4).

## 3. Assumption Inventory

### ASM-1: Manual Process Is the Primary Adoption Barrier

| Field | Value |
|-------|-------|
| Statement | The manual process of navigating kit files, sequencing artifacts, and tracking state is the primary barrier preventing AIEOS adoption by teams beyond the framework creator. |
| Source | PFD §5 Opportunity Sizing — "Removes the primary barrier to adoption by other teams and organizations" |
| Category | Market |
| Risk Level | High |
| Impact if False | If adoption is blocked by other factors (framework maturity, trust, organizational fit, content quality), a guided interface will not meaningfully increase adoption. The initiative solves the wrong problem. |
| Current Evidence | Direct observation from operating the framework (Known). No external validation — no other teams have attempted adoption, so the barrier has not been tested with external users. |
| Validation Method | Structured interviews with 3-5 potential adopters (engineering leads or product managers at other teams/orgs) to identify their top concerns about AIEOS adoption. Ask open-ended questions about adoption barriers before mentioning the manual process. |

### ASM-2: Non-Technical Users Want to Participate in AIEOS Processes

| Field | Value |
|-------|-------|
| Statement | Product managers and other non-technical stakeholders want to operate AIEOS discovery processes independently, rather than having a technical user drive the process on their behalf. |
| Source | PFD §3 UG-2 — "Cannot self-serve because operating the framework requires navigating markdown file structures"; VH HYP-2 — "non-technical users can operate independently" |
| Category | User Behavior |
| Risk Level | High |
| Impact if False | If non-technical users prefer to have a technical guide drive the process (and simply review/approve outputs), HYP-2 is invalid and the interface design for UG-2 is solving a problem they don't have. The console could still serve UG-1, but a major value proposition is lost. |
| Current Evidence | None — no non-technical users have been asked whether they want independent AIEOS operation. This is assumed based on general product management trends toward self-service tooling. |
| Validation Method | Interviews with 3-5 product managers (not currently AIEOS users) showing them the AIEOS discovery flow and asking: (a) would you want to drive this process yourself or have someone drive it for you? (b) what would make you confident enough to do it independently? |

### ASM-3: Four-File System Structure Is Stable Enough for Tooling

| Field | Value |
|-------|-------|
| Statement | The AIEOS four-file system (spec, template, prompt, validator) and kit directory structure are stable enough that tooling built against them will not be broken by near-term governance model changes. |
| Source | PFD §7 Current State — "well-defined, consistent structure across all 6 kits"; PFD §9 OQ-3 — "Is the four-file system structure stable enough to build tooling against?" |
| Category | Technical |
| Risk Level | Medium |
| Impact if False | If the governance model or kit structure changes significantly, the console would need to be updated to match. The more tightly the console depends on specific file layouts, the more costly changes become. |
| Current Evidence | The governance model is at v1.0 with no changes planned. Six kits have been built following the same structure. The structure has been stable through the entire kit development period (Known). |
| Validation Method | Review the governance model change protocol (§15) to assess what kinds of changes are permitted and how they would affect tooling. Assess whether the console can read kit structure dynamically rather than hardcoding artifact types. |

### ASM-4: AIEOS Prompts Produce Adequate Output Across LLM Providers

| Field | Value |
|-------|-------|
| Statement | The existing AIEOS prompts will produce artifacts of sufficient quality to pass validators when used with different LLM providers, not just the provider they were developed with. |
| Source | PFD §9 OQ-2 — "How well do existing AIEOS prompts perform across different LLM providers?"; VH HYP-3 — hypothesis that integrated validation improves quality assumes prompts produce reasonable first drafts |
| Category | Technical |
| Risk Level | High |
| Impact if False | If prompt output quality varies significantly across providers, "configurable LLM per artifact type" becomes a much harder problem — each provider may need prompt tuning. This could delay delivery and reduce the value of LLM configurability. |
| Current Evidence | Prompts were designed to be provider-agnostic (no provider-specific syntax). However, cross-provider testing has not been performed (Believed based on design intent, not validated). |
| Validation Method | Generate 3 artifact types (PFD, VH, AR) using at least 2 different LLM providers with the same inputs. Run validators on each output and compare pass rates and output quality. |

### ASM-5: A Guided Interface Is the Right Interaction Model

| Field | Value |
|-------|-------|
| Statement | A step-by-step guided interface is the right interaction model for AIEOS process execution, as opposed to alternative models such as chat-based interaction, IDE plugins, or CLI-only tooling. |
| Source | VH HYP-1 — "guided artifact sequencing"; VH HYP-2 — "guided interface that abstracts file-system navigation"; PFD §9 OQ-1 — "What interaction model best serves mixed technical and non-technical users?" |
| Category | User Behavior |
| Risk Level | Medium |
| Impact if False | If users prefer a different interaction model (e.g., conversational AI, IDE integration), the guided interface may feel rigid or cumbersome. Experienced users (UG-1) may find it too constraining; non-technical users (UG-2) may find a different model more natural. |
| Current Evidence | General UX patterns suggest step-based wizards work well for sequential, multi-step processes (Believed). No AIEOS-specific testing has been done. |
| Validation Method | Low-fidelity prototype or mockup of 2-3 interaction models (wizard, chat-based, hybrid) shown to 3-5 representative users from UG-1 and UG-2. Gather preference and usability impressions before committing to a model. |

### ASM-6: Validators Provide Sufficient Quality Signal

| Field | Value |
|-------|-------|
| Statement | AIEOS validators, when run as automated checks within the console, will provide a meaningful quality signal — i.e., PASS results correlate with actually-good artifacts and FAIL results correctly identify problems. |
| Source | VH HYP-3 — "integrating validator execution directly into the artifact generation workflow will achieve higher first-pass validation rates"; VH HYP-3 evidence criteria — "no artifacts are frozen without passing validation" |
| Category | Technical |
| Risk Level | Medium |
| Impact if False | If validators are too lenient (pass bad artifacts) or too strict (fail acceptable artifacts), the integrated validation loop either provides false confidence or creates frustrating rework cycles. Either outcome undermines HYP-3. |
| Current Evidence | Validators have been used in the current manual process and appear to provide useful quality signals (Known from direct experience). However, they have not been tested at scale or across varied content quality. |
| Validation Method | Run validators against a set of deliberately varied-quality artifacts (known-good, borderline, known-bad) and assess whether the PASS/FAIL results match human judgment. Minimum 5 artifacts per quality tier. |

### ASM-7: Single-User Operation Is Sufficient for Initial Adoption

| Field | Value |
|-------|-------|
| Statement | Teams evaluating or initially adopting AIEOS will accept a single-user console (with lock file for concurrency prevention) as a starting point, rather than requiring multi-user collaboration from day one. |
| Source | PFD §8 Constraints and Boundaries — "Single-user operation initially"; VH HYP-4 target users UG-3 — evaluators forming opinions about AIEOS fit |
| Category | Organizational |
| Risk Level | Low |
| Impact if False | If teams require multi-user collaboration from the start, the single-user console will not meet their minimum requirements. This delays adoption but does not invalidate the core hypotheses — it changes the delivery scope. |
| Current Evidence | Many development tools start single-user and add collaboration later (Believed based on industry patterns). No AIEOS-specific data. |
| Validation Method | Include in adopter interviews (ASM-1 validation): ask whether single-user operation would be acceptable for initial evaluation and early use. |

## 6. Dependency Map

### Inter-Assumption Dependencies

- **ASM-1 → ASM-2:** If the manual process is not the primary adoption barrier (ASM-1 false), then whether non-technical users want independent operation (ASM-2) becomes less relevant — other barriers must be addressed first.
- **ASM-4 → ASM-6:** If prompts don't produce adequate output across providers (ASM-4 false), then validator quality signal (ASM-6) may be moot — artifacts may fail validation due to poor generation rather than validator issues.
- **ASM-5 → ASM-2:** If the guided interface is not the right model (ASM-5 false), then non-technical independent operation (ASM-2) may require a different interaction approach entirely.

### Cascade Analysis

- **ASM-1 invalidation cascades to:** ASM-2, ASM-5, ASM-7 — if the manual process isn't the barrier, the specific interaction model, user independence, and single-user sufficiency assumptions all need re-examination.
- **ASM-4 invalidation cascades to:** ASM-6 — validator testing becomes secondary if the generation quality problem must be solved first.
